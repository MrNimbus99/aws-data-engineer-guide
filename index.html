Quiz 2 — Sample Question
Domain 1: Data Ingestion and Transformation
A company stores 5 TB of daily transaction logs as CSV files in Amazon S3. Analysts query these files using Amazon Athena but report high costs and slow performance. A data engineer needs to transform the data to reduce Athena scan costs. Which approach should the data engineer use?
  • Convert CSV to Parquet without compression and store as a single large file per day
  • Convert CSV to Avro using Glue for maximum Athena performance
  • ✓ Convert CSV to columnar Parquet using an AWS Glue ETL job
  • Keep the data as plain text CSV for simplicity
Show Explanation
✓ Why correctApache Parquet is a columnar storage format that supports predicate pushdown and column pruning. When Athena queries Parquet data, it reads only the columns and row groups needed, dramatically reducing the amount of data scanned. Since Athena charges per TB scanned, this directly lowers query cost and improves performance. AWS Glue natively supports reading CSV and writing Parquet via its DynamicFrame or Spark DataFrame APIs.
✗ Why A is wrongWhile converting to Parquet is correct, storing it uncompressed and as a single large file negates key performance benefits. A single large file prevents parallelized reads by Athena (which performs best with many appropriately sized files), and no compression means more bytes scanned per query — increasing both cost and latency.
✗ Why B is wrongWhile Avro is a valid format for serialization (especially for schema evolution in streaming), it is row-based. Athena queries on row-based formats scan more data than necessary for analytical queries that typically access a subset of columns.
✗ Why D is wrongPlain text (uncompressed CSV) is the worst option for Athena. It offers no columnar optimization, no compression, and no predicate pushdown, resulting in maximum data scanned and highest query cost.

Quiz 9 — Sample Question
Domain 2: Data Store Management
A retail company is building a star-schema analytics warehouse in Amazon Redshift. The central fact table contains 800 million order records, and the dimension tables (products, stores, customers) each contain fewer than 500,000 rows. Analysts primarily query orders filtered by date range and joined with dimension tables. Which schema design minimizes query execution time?
  • Apply KEY distribution on the fact table's surrogate primary key and AUTO distribution to dimension tables
  • Apply ALL distribution to the fact table and KEY distribution on each dimension table's primary key
  • Apply AUTO distribution to all tables and rely on Redshift Advisor to optimize over time
  • ✓ Apply EVEN distribution to the fact table and ALL distribution to dimension tables; use a compound sort key on the order date column of the fact table
Show Explanation
✓ Why D is correctEVEN distribution spreads the large fact table across all slices for balanced parallel processing. ALL distribution replicates the small dimension tables to every compute node, eliminating inter-node data movement during joins. A compound sort key on the order date column enables zone-map pruning, allowing Redshift to skip irrelevant disk blocks for date-range-filtered queries.
✗ Why A is wrongKEY distribution on a surrogate primary key (unique per row) distributes rows essentially randomly across slices, providing no join colocation benefit. AUTO distribution on small dimension tables may eventually choose ALL, but this leaves optimization to chance rather than design.
✗ Why B is wrongALL distribution on an 800-million-row fact table would replicate the entire table to every node, consuming massive storage and dramatically slowing data loads. ALL is appropriate only for small tables.
✗ Why C is wrongWhile AUTO distribution and Redshift Advisor can help, they react to observed query patterns over time. For a known star-schema workload at this scale, explicit distribution and sort key design yields superior performance from day one.

Domain 4: Data Security and Governance
A data engineering team runs an Amazon Redshift cluster in a private subnet. A new ETL application on Amazon EC2 in a separate subnet needs to query the cluster. The team wants to allow only the minimum required network access. Which approach should the team use?
  • Add an inbound rule to the Redshift security group allowing TCP port 5439 from 0.0.0.0/0
  • ✓ Add an inbound rule to the Redshift security group allowing TCP port 5439 from the EC2 instance's security group
  • Add a network ACL rule on the Redshift subnet allowing TCP port 5439 from the EC2 subnet CIDR block
  • Add an inbound rule to the Redshift security group allowing all TCP ports from the EC2 subnet CIDR block
Show Explanation
✓ Why correct (B)The team should add an inbound rule to the Redshift cluster's VPC security group that allows TCP port 5439 (Redshift's default port) from the EC2 instance's security group as the source. This follows the principle of least privilege at the network level because security group references allow only traffic originating from resources attached to the specified security group, without needing to hard-code IP addresses.
✗ Why A is wrongOpening port 5439 to 0.0.0.0/0 allows inbound traffic from any IP address on the internet. Even in a private subnet, this is overly permissive and violates least-privilege network access principles.
✗ Why C is wrongA network ACL (NACL) with the subnet CIDR block is broader than necessary because it permits all resources in the entire subnet, not just the ETL application. NACLs are also stateless, requiring separate inbound and outbound rules, making them harder to manage. Security groups are stateful and more granular.
✗ Why D is wrongOpening all ports (0-65535) from the EC2 subnet CIDR is far too permissive. Only port 5439 is needed for Redshift connections. This violates the principle of least privilege by exposing all ports on the cluster.

A company has multiple Amazon Redshift clusters across different AWS accounts within the same organization. The analytics team in one account needs read-only access to product catalog data stored in another account's Redshift cluster. The company wants to avoid data duplication and minimize operational overhead. Which approach should the data engineer implement?
  • ✓ Use Redshift data sharing to create a datashare from the producer cluster and grant usage to the consumer account's Redshift namespace
  • Set up an AWS Glue ETL job to copy the product catalog data nightly from the producer cluster to the consumer cluster
  • Create a cross-account IAM role and use Redshift Spectrum to query the producer cluster's data directly
  • Export the data to Amazon S3 and configure cross-account S3 bucket policies for the consumer account
Show Explanation
✓ Why A is correctAmazon Redshift data sharing allows you to share live data across Redshift clusters and AWS accounts without copying or moving data. The producer creates a datashare, adds schemas and tables, and grants usage to the consumer's Redshift namespace. This provides real-time read access with zero data duplication and minimal operational overhead.
✗ Why B is wrongRunning a nightly AWS Glue ETL job introduces data duplication, additional storage costs, pipeline maintenance burden, and data staleness — the consumer's data could be up to 24 hours old.

✗ Why D is wrongExporting to S3 requires building and maintaining an export pipeline, creates data duplication, and introduces latency. Redshift data sharing is the purpose-built, zero-copy solution for cross-account Redshift data access.

Domain 4: Data Security and Governance
A data engineering team has 15 analysts who all need read access to the same set of S3 data lake buckets and Amazon Athena query permissions. New analysts join frequently and existing analysts sometimes move to other teams. The team lead wants to minimize the operational overhead of managing these permissions. What is the MOST efficient approach?
  • Attach identical IAM policies to each analyst's IAM user individually
  • Use S3 bucket policies to grant access to each analyst's IAM user ARN
  • Create a single IAM role and share the role credentials among all analysts
  • ✓ Create an IAM group with the required policies and add all analysts to the group
Show Explanation
✓ Why correctCreating an IAM group with the required S3 and Athena policies attached, then adding/removing users from the group, is the most efficient approach. When a new analyst joins, you simply add them to the group and they inherit all permissions. When someone leaves, removing them from the group revokes access instantly. This avoids managing individual inline policies for each user.
✗ Why A is wrongAttaching identical policies to each IAM user individually creates significant management overhead. Every policy change must be replicated across all 15 users, and onboarding/offboarding requires manually adding or removing multiple policies per user.
✗ Why B is wrongS3 bucket policies can grant access but do not address Athena permissions, and managing access for 15+ users directly in bucket policies becomes unwieldy. Bucket policies are better suited for cross-account access or public/service-level rules, not per-user team management.
✗ Why C is wrongCreating a single IAM role and sharing credentials violates security best practices. IAM roles are assumed temporarily by trusted entities — they are not designed for permanent shared access by multiple human users. Each user should have their own IAM identity for auditability.

A data engineer needs to run ad-hoc SQL queries against CSV files stored in an S3 data lake. The data is partitioned by date, and the team wants to minimize both cost and operational overhead. Which approach should the data engineer use?
  • Load the CSV data into an Amazon Redshift cluster and run queries there
  • ✓ Use Amazon Athena with partitioned external tables defined in the AWS Glue Data Catalog

  • Set up Amazon OpenSearch Service to index and search the CSV data
Show Explanation
✓ Why correctAmazon Athena is serverless—there is no infrastructure to provision or manage. By defining partitioned external tables in the AWS Glue Data Catalog (e.g., partitioned by date), Athena scans only the relevant S3 prefixes, which dramatically reduces the amount of data scanned and therefore cost (Athena charges per TB scanned). This combination delivers the lowest operational overhead and cost for ad-hoc SQL queries on S3 data.
✗ Why A is wrongLoading data into Amazon Redshift requires provisioning and managing a cluster (or Redshift Serverless), which adds significant operational overhead compared to Athena for ad-hoc queries. Redshift is better suited for repeated, complex analytical workloads rather than occasional ad-hoc exploration.
✗ Why C is wrongAmazon EMR requires launching and managing a cluster, selecting instance types, and configuring Hive or Spark. This adds operational overhead that the scenario explicitly wants to avoid.
✗ Why D is wrongAmazon OpenSearch Service requires provisioning a domain, indexing all the data, and managing the cluster. It is designed for full-text search and log analytics, not ad-hoc SQL queries over CSV files in S3.

A data engineering team needs to share encrypted S3 objects with an analytics team in a different AWS account. The objects are encrypted using a customer-managed KMS key. The analytics team reports AccessDenied errors when trying to read the objects. What should the data engineering team do to resolve this?
  • Re-encrypt all objects using SSE-S3 before sharing, since KMS keys cannot work across accounts
  • ✓ Grant the analytics account's IAM role kms:Decrypt permission in the KMS key policy and ensure the S3 bucket policy allows cross-account access
  • Create a new KMS key in the analytics account and use S3 Batch Operations to re-encrypt all objects
  • Enable S3 Object Lambda to decrypt objects on-the-fly when the analytics team accesses them
Show Explanation
✓ Why correctTo share KMS-encrypted S3 objects cross-account, you must grant the external account's IAM principal kms:Decrypt (and optionally kms:DescribeKey) permission in the KMS key policy, AND allow cross-account access in the S3 bucket policy. Both permissions are required — KMS key policy for decryption and bucket policy for object access.
✗ Why A is wrongKMS customer-managed keys DO support cross-account access via key policies. Re-encrypting with SSE-S3 removes the ability to centrally manage and audit key usage through KMS.
✗ Why C is wrongCreating a new key and re-encrypting all objects is unnecessary operational overhead. KMS key policies natively support granting cross-account principals access to existing keys.
✗ Why D is wrongS3 Object Lambda is designed for transforming data on retrieval (e.g., redacting PII), not for handling cross-account KMS decryption. The issue is missing permissions, not a need for data transformation.

A data engineering team needs to trigger an AWS Glue ETL job automatically whenever new files arrive in an S3 bucket, and also trigger a separate Lambda function for data quality checks on the same event. Which TWO configurations should the team use to reliably fan out S3 object-creation events to multiple targets?
  • ✓ Enable Amazon EventBridge notifications on the S3 bucket to route events
  • Configure S3 Event Notifications to send directly to both Glue and Lambda simultaneously
  • Use Amazon SNS as an intermediary between S3 Event Notifications and the targets
  • ✓ Create separate EventBridge rules for the Glue job target and the Lambda target
  • Create a single EventBridge rule with both Glue and Lambda as targets in the same rule
Show Explanation
✓ Why correctAmazon EventBridge can receive S3 object-creation events (when enabled on the bucket) and route them to multiple targets using separate rules, making it ideal for fan-out scenarios. Configuring separate EventBridge rules for each target (Glue job and Lambda) allows independent filtering, retry logic, and error handling per target.
✓ Why A is correctEventBridge natively supports S3 event notifications and can route a single event to multiple targets via separate rules, which is the recommended approach for fan-out.
✗ Why B is wrongS3 Event Notifications can only send each event type to a single destination per prefix/suffix combination. To fan out to both Glue and Lambda, you would need an intermediary like SNS or EventBridge.
✗ Why C is wrongWhile SNS fan-out with S3 Event Notifications is a valid pattern, EventBridge provides richer filtering, built-in retry policies, and direct integration with Glue jobs as targets — making it the more operationally efficient choice for this scenario.
✓ Why D is correctSeparate EventBridge rules for each target ensure independent event processing, retry configuration, and dead-letter queue assignment per target.
✗ Why E is wrongA single EventBridge rule with multiple targets is technically possible (up to 5 targets per rule), and each target does get independent retry/DLQ configuration. However, separate rules per target is the AWS best practice because it provides independent rule-level monitoring (each rule has its own CloudWatch metrics for invocations and failures), independent enable/disable control, and clearer operational separation. With a single rule, you cannot disable or modify one target's routing without affecting the other.

A data engineering team runs nightly AWS Glue ETL jobs that load data into Amazon Redshift. Occasionally jobs fail silently, and the team only discovers the failure the next morning when analysts report missing data. The team wants to be alerted within minutes of a job failure. Which approach should they implement?
  • ✓ Create a CloudWatch alarm on Glue job failure metrics and configure an SNS notification to alert the team
  • Enable AWS Config rules to detect Glue job failures and send notifications
  • Use AWS CloudTrail to monitor Glue job execution status and trigger alerts on failure
  • Configure AWS Trusted Advisor to monitor Glue job health and notify the team of failures
Show Explanation
✓ Why correct (A)Amazon CloudWatch receives Glue job state-change events automatically. Creating a CloudWatch alarm on the glue.driver.aggregate.numFailedTasks metric, or using an EventBridge rule on Glue job state changes routed to an SNS topic, enables near-real-time alerting when a job fails. However, the most direct approach is a CloudWatch alarm on the Glue job's built-in failure metrics, which triggers an SNS notification to the team.
✗ Why B is wrongAWS Config tracks resource configuration changes (e.g., whether a Glue job's settings changed), not runtime execution outcomes like job success or failure. Config cannot detect that a specific job run failed.
✗ Why C is wrongCloudTrail logs API calls (e.g., StartJobRun, CreateJob) but does not track the runtime status or outcome of a Glue job execution. A job can be started successfully (logged by CloudTrail) and still fail during processing.
✗ Why D is wrongTrusted Advisor provides account-level best-practice checks around cost optimization, security, fault tolerance, and service limits. It does not monitor individual job executions or provide failure alerting for ETL pipelines.

Domain 1: Data Ingestion and Transformation
A company is migrating a 2 TB Oracle database from on-premises to Amazon RDS for PostgreSQL. The application must remain available during migration with less than 1 hour of downtime. Which approach meets these requirements?
  • Use AWS DMS with full load only and cut over immediately after the load completes
  • ✓ Use AWS DMS with full load plus change data capture (CDC) to replicate ongoing changes during migration
  • Export the database to CSV files, upload to S3, and use COPY to load into the target RDS instance
  • Use AWS Backup to create a snapshot of the on-premises database and restore it in RDS
Show Explanation
✓ Why correct (B)AWS DMS with full load plus CDC first copies the entire database (full load), then continuously replicates ongoing changes (CDC) from the source to the target. This allows the application to keep running on the source database during migration. When the target is caught up, you perform a brief cutover with minimal downtime — well under 1 hour for most workloads.
✗ Why A is wrongFull load only copies data at a point in time but does not capture changes made during the migration. For a 2 TB database, the full load could take hours, during which new transactions would be lost, requiring extended downtime.
✗ Why C is wrongExporting to CSV and reloading is a manual, time-consuming process that does not handle ongoing changes. It would require the application to be offline for the entire export-upload-load cycle, resulting in significant downtime.
✗ Why D is wrongAWS Backup does not support backing up on-premises databases. It is designed for AWS-native resources such as RDS instances, EBS volumes, and DynamoDB tables.

A company runs an ETL pipeline using AWS Glue in a VPC with no internet gateway. The pipeline reads raw data from Amazon S3 and writes transformed data back to S3. The security team requires that all traffic between the VPC and AWS services stays off the public internet. Which solution meets this requirement?
  • Configure a NAT gateway in the VPC and route traffic to the S3 and Glue public endpoints
  • Create a VPC peering connection to the AWS-managed VPC that hosts Amazon S3 and AWS Glue
  • ✓ Create an interface VPC endpoint powered by AWS PrivateLink for the Glue API and a gateway VPC endpoint for Amazon S3
  • Set up an AWS Site-to-Site VPN connection between the VPC and the AWS service endpoints
Show Explanation
✓ Why C is correctAWS PrivateLink enables private connectivity to AWS services through interface VPC endpoints. For AWS Glue, an interface VPC endpoint (powered by PrivateLink) allows Glue jobs to reach the Glue API without traversing the internet. For Amazon S3, a gateway VPC endpoint routes S3 traffic privately through the AWS network at no additional cost. Together, these keep all ETL traffic off the public internet.
✗ Why A is wrongA NAT gateway routes traffic out through the internet (via an internet gateway), which violates the requirement to keep traffic off the public internet. It also adds NAT gateway data processing charges.
✗ Why B is wrongYou cannot create a VPC peering connection to AWS-managed service VPCs. VPC peering connects two customer-owned VPCs, not a customer VPC to an internal AWS service network.
✗ Why D is wrongSite-to-Site VPN is designed for connecting on-premises networks to AWS over an encrypted tunnel. It is not used for private connectivity between a customer VPC and AWS services within the same region.

A company receives events from multiple third-party SaaS applications (e.g., Zendesk, Datadog, Auth0). A data engineer needs to build a pipeline that routes these events to different downstream AWS services based on event type with minimal custom code. Which approach meets these requirements?

  • Use Amazon Kinesis Data Streams to ingest events from each SaaS application using custom producer code

  • Use AWS Lambda functions with scheduled polling to check each SaaS provider's API for new events every minute
Show Explanation
✓ Why correct (C)Amazon EventBridge is the best fit here because it natively integrates with SaaS partners and AWS services, can receive events via rules from multiple sources, and route them to different targets (Lambda, Step Functions, SNS) based on event content. This decouples the SaaS applications from downstream processing and lets the team add new event consumers without modifying producers.
✗ Why A is wrongAmazon SNS with SQS provides pub/sub messaging but lacks native SaaS partner integrations. The team would need to build custom webhook receivers and manage message filtering manually, which increases operational overhead.
✗ Why B is wrongAmazon Kinesis Data Streams is designed for high-throughput, ordered streaming data ingestion (e.g., clickstreams, IoT). It requires custom producers/consumers and does not have built-in SaaS partner event integrations or content-based routing rules.
✗ Why D is wrongAWS Lambda with scheduled polling would require writing and maintaining custom code to poll each SaaS API on a schedule. This introduces latency (not real-time), increases costs from unnecessary invocations when no events exist, and creates tight coupling to each SaaS provider's API.
